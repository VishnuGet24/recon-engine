## 1. Project Overview

### 1.1 Application Purpose
SF Recon Engine is a security reconnaissance dashboard application designed for security professionals to monitor, manage, and analyze security assets, vulnerabilities, and scan operations. The application provides:

- **Real-time monitoring** of security posture through metrics and dashboards
- **Asset inventory management** with comprehensive filtering and search capabilities
- **Scan configuration and execution** for security reconnaissance operations
- **Finding and vulnerability tracking** across all discovered assets

### 1.2 Frontend Role
The frontend serves as the **presentation and interaction layer** that:
- Displays real-time security metrics and trends
- Provides interactive interfaces for asset management
- Enables scan configuration and monitoring
- Visualizes security data through charts and tables
- Manages user interactions and state

### 1.3 Architecture Pattern
The frontend follows a **client-side rendered SPA (Single Page Application)** architecture using:
- React Router for declarative routing
- Component-based architecture for modularity
- RESTful API consumption for data operations
- WebSocket connections for real-time updates (future implementation)

## 3. Environment Setup

### 3.1 Local Development Installation

#### Step 1: Clone and Install
```bash
# Clone the repository
git clone <repository-url>
cd sf-recon-engine

# Install dependencies
pnpm install
# or
npm install
```

#### Step 2: Configure Environment Variables
Create a `.env` file in the project root:

```env
# API Configuration
VITE_API_BASE_URL=http://localhost:8000/api
VITE_API_TIMEOUT=30000

# WebSocket Configuration
VITE_WS_URL=ws://localhost:8000/ws

# Authentication
VITE_AUTH_TOKEN_KEY=sf_recon_auth_token
VITE_REFRESH_TOKEN_KEY=sf_recon_refresh_token

# Feature Flags
VITE_ENABLE_WEBSOCKET=true
VITE_ENABLE_REAL_TIME_UPDATES=true

# Environment
VITE_ENV=development
```

**Important Notes:**
- All Vite environment variables MUST be prefixed with `VITE_`
- Variables are exposed to the client-side code
- **Never commit `.env` files with production credentials**

#### Step 3: Start Development Server
```bash
pnpm build
# Development server typically runs on http://localhost:5173
```

### 3.2 Environment Variable Usage
Access environment variables in React components:

```typescript
// src/config/api.ts
export const API_CONFIG = {
  baseURL: import.meta.env.VITE_API_BASE_URL || 'http://localhost:8000/api',
  timeout: parseInt(import.meta.env.VITE_API_TIMEOUT) || 30000,
  wsURL: import.meta.env.VITE_WS_URL || 'ws://localhost:8000/ws',
};
```

### 3.3 CORS Configuration
The backend MUST configure CORS to allow requests from the frontend origin:

```javascript
// Example Express.js CORS config
const cors = require('cors');
app.use(cors({
  origin: process.env.FRONTEND_URL || 'http://localhost:5173',
  credentials: true,
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-Requested-With'],
}));
```

## 7. Error Handling

### 7.1 Standard Error Response Format
All error responses MUST follow this format:

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error message",
    "details": [
      {
        "field": "fieldName",
        "message": "Field-specific error message"
      }
    ],
    "timestamp": "2026-03-11T12:30:00.000Z",
    "requestId": "req-12345"
  }
}
```

### 7.2 HTTP Status Codes
| Status Code | Meaning | Frontend Action |
|-------------|---------|-----------------|
| 200 | Success | Process response data |
| 201 | Created | Show success message, redirect |
| 400 | Bad Request | Display validation errors |
| 401 | Unauthorized | Attempt token refresh, or redirect to login |
| 403 | Forbidden | Show "Insufficient permissions" message |
| 404 | Not Found | Show "Resource not found" message |
| 409 | Conflict | Show conflict message (e.g., scan already running) |
| 422 | Validation Error | Display field-specific errors |
| 429 | Rate Limit | Show "Too many requests, please wait" |
| 500 | Server Error | Show generic error, log to monitoring |
| 503 | Service Unavailable | Show "Service temporarily unavailable" |

### 7.3 Frontend Error Handling Strategy

#### 7.3.1 Global Error Handler
```typescript
// src/utils/errorHandler.ts
export const handleApiError = (error: any) => {
  if (!error.response) {
    // Network error
    toast.error('Network error. Please check your connection.');
    return;
  }
  
  const { status, data } = error.response;
  
  switch (status) {
    case 400:
      toast.error(data.error?.message || 'Invalid request');
      break;
      
    case 401:
      // Handled by axios interceptor (token refresh)
      break;
      
    case 403:
      toast.error('You do not have permission to perform this action');
      break;
      
    case 404:
      toast.error('Resource not found');
      break;
      
    case 409:
      toast.error(data.error?.message || 'Conflict occurred');
      break;
      
    case 422:
      // Validation errors - handled by form components
      if (data.error?.details) {
        data.error.details.forEach((detail: any) => {
          toast.error(`${detail.field}: ${detail.message}`);
        });
      }
      break;
      
    case 429:
      toast.error('Too many requests. Please wait a moment.');
      break;
      
    case 500:
    case 503:
      toast.error('Server error. Please try again later.');
      // Log to monitoring service
      console.error('Server error:', data);
      break;
      
    default:
      toast.error('An unexpected error occurred');
  }
};
```

const apiClient = axios.create({

// Request interceptor - Add auth token
apiClient.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('sf_recon_auth_token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// Response interceptor - Handle token refresh
apiClient.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    
    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true;

        const refreshToken = localStorage.getItem('sf_recon_refresh_token');
        const response = await axios.post(`${API_CONFIG.baseURL}/auth/refresh`, {
          refreshToken
        });
        
        localStorage.setItem('sf_recon_auth_token', response.data.accessToken);
        originalRequest.headers.Authorization = `Bearer ${response.data.accessToken}`;
        
        return apiClient(originalRequest);

        localStorage.clear();
        window.location.href = '/login';
        return Promise.reject(refreshError);
      }
    }
    
    return Promise.reject(error);
  }
);

### API Endpoints Summary

|----------|--------|---------|
| `/api/auth/login` | POST | User login |

### HTTP Status Codes
- `200` - Success
- `201` - Created
- `400` - Bad Request
- `401` - Unauthorized (token refresh or login required)
- `403` - Forbidden
- `404` - Not Found
- `409` - Conflict
- `422` - Validation Error
- `429` - Rate Limited
- `500` - Server Error
- `503` - Service Unavailable
